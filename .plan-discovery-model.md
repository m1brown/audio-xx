# Discovery-First Conversation Model — Proposed Design

## Problem

The current engine reaches recommendations too quickly. Shopping intent detection, question sequencing, and answer-readiness are spread across three files (`clarification.ts`, `shopping-intent.ts`, `page.tsx`) with overlapping turn-cap logic. The system can jump to a recommendation after as few as 1–2 turns if keyword matches happen to satisfy the boolean flags in `isAnswerReady()`.

The root issue: context flags like `tasteProvided` and `systemProvided` are binary keyword matches. A user who casually mentions "my speakers" gets `systemProvided: true` even though the system knows almost nothing about their chain. The engine treats presence as sufficiency.

---

## Proposed: DiscoveryState

A single structured object that tracks what the system actually knows — not just whether a keyword appeared, but what was extracted.

```typescript
interface DiscoveryState {
  /** What the user wants to change or improve. */
  goal: {
    gathered: boolean;
    raw: string | null;          // their words
    category: ShoppingCategory;  // extracted: dac, amplifier, speakers, etc.
    mode: ShoppingMode;          // specific-component, upgrade-path, build-a-system
  };

  /** Taste and listening preferences. */
  preferences: {
    gathered: boolean;
    traits: Record<string, SignalDirection>;  // from signal engine
    symptomCount: number;                     // how many symptoms matched
    disambiguated: boolean;                   // ambiguous phrases resolved?
  };

  /** Existing system context. */
  system: {
    gathered: boolean;
    profile: SystemProfile;      // output type, character, tube, low-power
    described: boolean;          // did the user describe actual components?
  };

  /** Budget. */
  budget: {
    gathered: boolean;
    amount: number | null;
    openToUsed: boolean | null;  // null = not asked yet
  };

  /** Conversation metadata. */
  turnCount: number;
  phase: 'discovery' | 'ready' | 'answered';
}
```

### Key differences from today

| Today (ShoppingContext)                | Proposed (DiscoveryState)                        |
|----------------------------------------|--------------------------------------------------|
| `tasteProvided: boolean` (≥2 symptoms) | `preferences.gathered` + `symptomCount` + `disambiguated` |
| `systemProvided: boolean` (keyword)    | `system.gathered` + `system.described` + full `SystemProfile` |
| `budgetMentioned: boolean` (regex)     | `budget.gathered` + `budget.amount` + `budget.openToUsed` |
| No goal tracking                       | `goal.gathered` + `goal.raw` + extracted category/mode |
| Turn cap is the only brake             | Phase transitions require minimum evidence        |

## Question Selection Logic

The system maintains a priority-ordered question sequence. On each turn, it walks the list and asks the first question whose field is not yet adequately gathered.

### Priority order (all modes)

```
1. Goal         — if goal.gathered is false
2. Preferences  — if preferences.symptomCount < 2
3. System       — if system.described is false
4. Budget       — if budget.gathered is false
5. Used market  — if budget.openToUsed is null AND budget.amount is known
```

Steps 1–4 are the core discovery loop. Step 5 is conditional — only asked when budget is established and the answer might materially affect the recommendation (e.g., budget is tight).

### Question templates

Questions are category-aware, same as today. The goal question is new:

```
Goal questions (by mode):
  specific-component: "What are you hoping a new [category] will change about your listening experience?"
  upgrade-path:       "What feels most limiting about what you hear right now?"
  build-a-system:     "What kind of listening experience are you trying to build toward?"

Preference questions:  (same as current TASTE_QUESTIONS, per category)
System questions:      (same as current SYSTEM_QUESTIONS, per category)
Budget question:       "Do you have a budget range in mind?"
Used market question:  "Are you open to used or ex-demo gear, or prefer buying new?"
```

### Disambiguation pass

After preferences are gathered (symptomCount ≥ 2), the system checks for ambiguous phrases (the existing `AMBIGUOUS_PHRASES` list in `clarification.ts`). If an ambiguous phrase is found and `disambiguated` is false, it inserts a disambiguation question before moving to the next field. This replaces the current approach where disambiguation competes with shopping questions in priority order.

## Phase Transitions

```
discovery → ready:
  All four core fields gathered:
    goal.gathered        = true
    preferences.gathered = true  (symptomCount ≥ 2, disambiguated or no ambiguity)
    system.gathered      = true
    budget.gathered      = true

  OR turn cap reached (5 turns — raised from 3)

ready → answered:
  buildShoppingAnswer() called, answer dispatched
```

When `phase` transitions to `ready` via turn cap (incomplete context), the answer builder works with what it has. The answer can note which context was missing and what assumptions it made.

---

## Turn cap

Raised from 3 to 5. The turn cap is a safety valve, not the primary control. In practice, most conversations should reach `ready` in 3–4 turns because users often provide multiple signals per message. The extra headroom prevents the system from cutting discovery short when a user gives partial answers.

## Where this lives

### New file: `apps/web/src/lib/discovery.ts`

Owns:
- `DiscoveryState` type
- `createInitialDiscoveryState()` — factory
- `updateDiscoveryState(state, userText, signals)` — rebuilds state from all user text + signals
- `getNextQuestion(state)` — returns the next discovery question or null
- `isDiscoveryComplete(state)` — checks phase transition readiness

### Modified files

| File | Change |
|------|--------|
| `conversation-types.ts` | Add `discoveryState: DiscoveryState` to `ConversationState` |
| `page.tsx` | Replace `getClarificationQuestion` call with `updateDiscoveryState` + `getNextQuestion`. Route to answer when discovery is complete. |
| `clarification.ts` | Disambiguation logic moves into `discovery.ts`. File becomes thin or is removed. |
| `shopping-intent.ts` | `detectShoppingIntent` still handles mode/category detection. `getShoppingClarification` and `isAnswerReady` are replaced by discovery logic. `buildShoppingAnswer` remains, but receives `DiscoveryState` instead of `ShoppingContext`. |

### Files NOT changed

| File | Why |
|------|-----|
| `product-scoring.ts` | Scoring is downstream of discovery — no changes needed |
| `products/dacs.ts` | Product catalog unchanged |
| `system-profile.ts` | Types unchanged, reused by DiscoveryState |
| `signal-types.ts` | Signal extraction unchanged |
| Rule engine / YAML | No changes — signals are still extracted the same way |

## Reducer integration

The discovery state is updated on every user turn:

```
handleSubmit:
  1. Dispatch ADD_USER_MESSAGE
  2. Fetch /api/evaluate → get signals + result
  3. Call updateDiscoveryState(currentDiscoveryState, allUserText, signals)
  4. If getNextQuestion(discoveryState) returns a question:
       → Dispatch ADD_QUESTION
  5. Else (discovery complete):
       → Build shopping answer from discoveryState
       → Dispatch ADD_SHOPPING_ANSWER
```

Discovery state persists across turns in the reducer (unlike today, where `ShoppingContext` is rebuilt from scratch on every turn). The `updateDiscoveryState` function is still deterministic — it re-scans all user text — but it also merges with the existing state so that explicit "gathered" flags aren't lost.

---

## What this does NOT change

- The signal engine (phrase matching, trait extraction)
- The rule engine (YAML rules, evaluation)
- Product scoring (trait alignment, budget fit, architecture affinity, system coherence)
- The product catalog
- Non-shopping conversations (diagnostic/inquiry flows still work as before)
- The advisor-first answer structure (ShoppingAnswer)

---

## Migration path

The existing `ShoppingContext` fields map directly to `DiscoveryState`:

```
ShoppingContext.detected       → goal.gathered (+ intent keywords)
ShoppingContext.mode           → goal.mode
ShoppingContext.category       → goal.category
ShoppingContext.tasteProvided  → preferences.gathered
ShoppingContext.systemProvided → system.gathered
ShoppingContext.budgetMentioned → budget.gathered
ShoppingContext.budgetAmount   → budget.amount
ShoppingContext.systemProfile  → system.profile
```

`buildShoppingAnswer` can accept `DiscoveryState` and extract what it needs, or we can build a `ShoppingContext`-shaped object from `DiscoveryState` to avoid touching the answer builder in the first pass.
